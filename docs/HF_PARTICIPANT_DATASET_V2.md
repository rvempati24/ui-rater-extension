# Hugging Face Participant Dataset v2

## Decision

The target layout for [`uxBench/ux-task-trace`](https://huggingface.co/datasets/uxBench/ux-task-trace) is participant-first. It replaces the earlier `model/website/run/attempt/user/session` hierarchy once participant management v2 is implemented.

Website and generator model remain important searchable metadata, but they do not own the trace. The stable ownership chain is:

```text
participant -> run -> task assignment -> attempt -> session artifacts
```

This matches the local participant-management model and keeps all rounds and retries for one participant together.

## Repository layout

```text
dataset-info.json
index/
  participants.jsonl
  runs.jsonl
  attempts.jsonl
participants/
  <participant-id>/
    participant.json
    runs/
      <run-id>/
        run.json
        tasks/
          <task-position>-<assignment-id>/
            task.json
            attempts/
              <attempt-number>-<attempt-id>/
                attempt.json
                manifest.json
                trace.json
                recording.webm
                snapshots/
                  s0001.jpg
                  s0001.json
                analysis/
                  input.json
                  findings.json
                  report.md
```

IDs are immutable, URL/path-safe identifiers generated by the server. Human-readable website names and task prompts belong in metadata, not directory names. Zero-padded task positions and attempt numbers make browser listings deterministic; IDs prevent collisions.

## Metadata files

### `participant.json`

Contains only stable study metadata that is intentionally exportable:

```json
{
  "schema_version": "2.0",
  "participant_id": "P004",
  "status": "active",
  "created_at": "2026-07-17T12:00:00Z",
  "metadata": {}
}
```

Operator notes, authentication data, and other local-only fields must not be exported. Although the current benchmark uses synthetic websites, the allow-list of exportable participant fields should remain explicit.

### `run.json`

Freezes the configuration for one participant round:

- `run_id`, `participant_id`, `study_id`, and lifecycle status;
- selected task source positions/order and random seed;
- website repository, revision, commit SHA, generator model, site, website run ID, and source URL;
- launcher configuration and timestamps;
- counts for assignments and accepted/invalidated/failed attempts.

Website provenance therefore remains available without controlling the directory hierarchy.

### `task.json`

Stores `assignment_id`, task order, source task ID/position, frozen task text, start URL, task origin such as Mind2Web, and accepted attempt ID when present.

### `attempt.json`

Stores the management state around one immutable recording:

```json
{
  "schema_version": "2.0",
  "participant_id": "P004",
  "run_id": "run_01...",
  "assignment_id": "asg_01...",
  "attempt_id": "att_01...",
  "attempt_number": 2,
  "session_id": "session_01...",
  "status": "accepted",
  "reason": null,
  "started_at": "...",
  "completed_at": "...",
  "artifact_checksums": {
    "manifest.json": "sha256:...",
    "trace.json": "sha256:...",
    "recording.webm": "sha256:..."
  }
}
```

`manifest.json`, `trace.json`, snapshots, video, and analysis outputs remain the canonical evidence. `attempt.json` connects that evidence to the participant-management lifecycle.

## Query indexes

The nested layout is easy to audit but expensive to scan. Root-level JSONL indexes provide one record per entity:

- `index/participants.jsonl`: one row per exported participant;
- `index/runs.jsonl`: one row per run, including website/model provenance;
- `index/attempts.jsonl`: one flattened row per attempt, including participant, run, task, status, artifact path, counts, and checksums.

The attempt index duplicates common query fields such as `generator_model`, `website`, `task_origin`, and `accepted`. This lets Hugging Face Dataset Viewer and downstream scripts filter without opening every nested metadata file. Nested metadata remains authoritative if an index needs rebuilding.

## Export eligibility and status changes

The safe default is to upload only completed runs and their accepted attempts. This avoids publishing a software-failure attempt before the operator has reviewed it.

Exporter modes should be explicit:

- `accepted` (default): completed runs, accepted attempts, and required parent metadata;
- `audit`: all completed, failed, and invalidated attempts with their current status/reason;
- `draft`: active runs for controlled debugging; never the default for HF upload.

If an accepted attempt is invalidated after it has been uploaded, the next synchronization updates `attempt.json` and the index row to `invalidated`; it does not silently erase the evidence. Consumers must filter `status == "accepted"` for the analysis-ready view. Permanent remote deletion is a separate confirmed maintenance operation.

## Upload unit and idempotency

One HF commit should synchronize one complete local run:

1. Read the run and attempts in one SQLite snapshot.
2. Validate IDs, statuses, referenced files, counts, and checksums.
3. Stage the participant/run tree and rebuild affected index rows.
4. Compare `(attempt_id, checksum)` with the remote index.
5. Upload new/changed files in one Hugging Face commit.
6. Record the returned commit SHA and upload time locally.

Re-running the exporter is idempotent: an unchanged attempt produces no artifact change. Reusing an existing `attempt_id` with different immutable evidence is an error unless an explicit repair workflow is used.

## Local-to-HF mapping

| Local entity/artifact | Hugging Face destination |
| --- | --- |
| SQLite participant row | `participants/<participant-id>/participant.json` |
| SQLite run row and frozen config | `participants/<participant-id>/runs/<run-id>/run.json` |
| SQLite task assignment | `.../tasks/<position>-<assignment-id>/task.json` |
| SQLite task attempt | `.../attempts/<number>-<attempt-id>/attempt.json` |
| `data/sessions/<session-id>/manifest.json` | same attempt directory |
| `trace.json`, snapshots, analysis | same attempt directory, preserving subfolders |
| associated WebM recording | `recording.webm` in the attempt directory |

The uploader resolves paths exclusively from local IDs and database relationships. User-controlled task text, website names, and filesystem paths never become remote path segments.

## Configuration

The existing local-retention and opt-in upload controls remain. Participant v2 adds:

```json
{
  "schema_version": "2.0",
  "layout": "participant-v2",
  "export_mode": "accepted",
  "sessions_dir": "data/sessions",
  "keep_local_export": true,
  "local_export_dir": "exports/ux-task-trace",
  "upload_hf": false,
  "hf_repo_id": "uxBench/ux-task-trace",
  "hf_revision": "participant-v2"
}
```

`upload_hf` remains false by default. `hf_revision` should initially be a dedicated branch so the participant layout can be validated before replacing the dataset's default revision.

## Migration from the legacy layout

Do not rearrange the existing remote files in place.

1. Import local `results.json` into participant-management v2 as documented in `PARTICIPANT_MANAGEMENT_V2.md`.
2. Generate stable legacy `run_id`, `assignment_id`, and `attempt_id` values and retain each original export path in migration metadata.
3. Export the participant layout to the `participant-v2` HF revision.
4. Verify entity counts, session IDs, artifact checksums, accepted status, and website provenance against the legacy export.
5. Freeze the legacy hierarchy as read-only and switch the dataset default revision only after consumers pass validation.

## Required validation

- Every attempt path starts under its matching participant and run.
- Every attempt references existing participant, run, assignment, and session IDs.
- At most one attempt per assignment has `status: accepted`.
- All accepted attempts have a manifest, trace, non-empty video, and referenced screenshots.
- Recorded counts agree with trace/snapshot contents.
- Checksums match staged files.
- Index rows point to existing nested metadata and contain no duplicate IDs.
- Website repository/revision/commit metadata survives the new participant-first layout.
- A second unchanged export produces no content diff.
